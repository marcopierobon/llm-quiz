In this lesson, you’ll meet the dataset we’ll be using this week — SAMSum, a benchmark for dialogue summarization — and see how well a base model like Llama 3.2 1B Instruct performs before any fine-tuning.


You’ll load the dataset, run a small evaluation, and record your model’s initial ROUGE score. That number becomes your reference point — the “before” snapshot that every fine-tuning experiment builds on.


By the end, you’ll understand the task, the metric, and the performance gap we’re aiming to close — the foundation for everything that follows in your first complete fine-tuning project.

Your First Fine-Tuning Project: SAMSum for Dialogue Summarization
This week is hands-on. You’re not just learning about fine-tuning — you’re doing it.

And like any good experiment, we start by asking:
"What's our baseline performance - what are we trying to beat?"

The SAMSum dataset is a collection of real-world chat conversations paired with human-written summaries. Each example looks like this:

dialogue:
Amanda: I baked cookies. Do you want some?
Jerry: Sure!
Amanda: I'll bring you tomorrow :-)

summary:
Amanda baked cookies and will bring Jerry some tomorrow.
The model’s task is to take a dialogue as input and generate a concise summary in a similar style.

This makes SAMSum ideal for our project because it’s:

Concrete and measurable — we can use the ROUGE metric to evaluate summary quality.
Short and efficient — runs fast enough for Colab experiments.
Realistic — mirrors real conversational summarization use cases.
Why SAMSum Works for This Exercise
We chose SAMSum because it’s small enough to experiment quickly, but rich enough to teach meaningful lessons about fine-tuning.

It’s a supervised summarization task, where improvement is easy to see: higher ROUGE scores mean better summaries.
And unlike math reasoning datasets, SAMSum measures generation quality — making it a perfect fit for exploring fine-tuning mechanics without needing reinforcement learning methods like PPO or GRPO.

Choosing the Base Model
For this week, our goal isn’t to build the best-performing math model — it’s to learn the mechanics of fine-tuning.

That means we’ll focus on a setup that’s small, fast, and fully transparent, so you can understand every step of the process.

We’ll use Llama 3 1B Instruct, Meta’s smallest open-weight model in the Llama 3 family.

Why this one?

It’s light enough to run comfortably in Google Colab, even without premium GPUs.
It supports instruction-following, so it can understand and respond to math-style prompts.
It’s large enough to show measurable improvements from fine-tuning — yet small enough for experiments to finish in a reasonable time.
In the real world, your choice of model would depend on your task, data scale, and performance needs — you might pick a 7B, 13B, or even a frontier model for enterprise-grade use cases.

But here, smaller is better. We’re learning the workflow — and everything you do with this 1B model scales directly to larger ones later.

This will be our “before” model — the reference point against which you’ll measure every improvement through the rest of the week.

Prerequisites — Dependencies
Everything in this baseline runs on open-source libraries available through pip:

pip install -q transformers datasets evaluate accelerate peft bitsandbytes
Here’s what each does:

transformers — provides the Llama 3.2 model, tokenizer, and generation pipeline.
datasets — handles SAMSum data loading, caching, and sampling.
evaluate — computes ROUGE and other metrics.
accelerate — enables efficient multi-device inference (used implicitly).
peft — supports LoRA/QLoRA (not used here yet, but ready for next lessons).
bitsandbytes — optional library for 4-bit quantization (we’ll use it later).
With these installed, you can reproduce the entire pipeline in Colab or your local environment.

Step 1 — Dataset Configuration
We define the dataset inside the project config.yaml:

dataset:
  name: knkarthick/samsum
  cache_dir: ../data/datasets
  field_map:
    input: dialogue
    output: summary
  type: completion
  splits:
    train: 200
    validation: 200
    test: 200
  seed: 42
This ensures reproducibility and keeps key configuration in one place. We will build upon this configuration file over the course of the week.

Step 2 — Load and Prepare the Dataset
Our helper function handles downloading, caching, and sampling:

def load_and_prepare_dataset(cfg):
    """
    Load dataset splits according to configuration.
    """
    cfg_dataset = cfg["dataset"]
    dataset = load_dataset(cfg_dataset["name"])
    val_key = "validation" if "validation" in dataset else "val"

    train = select_subset(dataset["train"], cfg_dataset["splits"]["train"], seed=42)
    val = select_subset(dataset[val_key], cfg_dataset["splits"]["validation"], seed=42)
    test = select_subset(dataset["test"], cfg_dataset["splits"]["test"], seed=42)

    print(f"📊 Loaded {len(train)} train / {len(val)} val / {len(test)} test samples.")
    return train, val, test
This function gives us exactly 200 samples per split for quick baseline testing.

Step 3 — Load the Model and Tokenizer
We use a unified setup function that supports both quantized and LoRA modes, but here we keep it simple:

def setup_model_and_tokenizer(cfg, use_4bit=False, use_lora=False):
    model_name = cfg["base_model"]
    tokenizer = AutoTokenizer.from_pretrained(model_name)
    tokenizer.pad_token = tokenizer.eos_token
    tokenizer.padding_side = "right"

    model = AutoModelForCausalLM.from_pretrained(
        model_name,
        device_map="auto",
        dtype=torch.bfloat16 if torch.cuda.is_available() else torch.float32,
    )

    print("🔹 Using base model only (no LoRA).")
    return model, tokenizer
Step 4 — Generate Predictions
Once loaded, we can prompt the model to summarize each dialogue:

def generate_predictions(model, tokenizer, dataset, task_instruction, batch_size=8):
    prompts = []
    for sample in dataset:
        user_prompt = (
            f"{task_instruction}\n\n"
            f"## Dialogue:\n{sample['dialogue']}\n## Summary:"
        )
        messages = [{"role": "user", "content": user_prompt}]
        prompt = tokenizer.apply_chat_template(messages, tokenize=False, add_generation_prompt=True)
        prompts.append(prompt)

    pipe = pipeline("text-generation", model=model, tokenizer=tokenizer, do_sample=False)
    preds = []
    for i in tqdm(range(0, len(prompts), batch_size), desc="Generating summaries"):
        batch = prompts[i : i + batch_size]
        outputs = pipe(batch, max_new_tokens=256, return_full_text=False)
        preds.extend([o[0]["generated_text"].strip() for o in outputs])

    return preds
The instruction used:

"You are a helpful assistant who writes concise, factual summaries of conversations. "
"Summarize the following conversation into a single sentence."
Step 5 — Compute ROUGE Scores
We use the evaluate package to calculate standard metrics:

def compute_rouge(predictions, samples):
    rouge = evaluate.load("rouge")
    references = [s["summary"] for s in samples]
    return rouge.compute(predictions=predictions, references=references)
What is ROUGE?
ROUGE (Recall-Oriented Understudy for Gisting Evaluation) is a standard metric for evaluating text summarization quality.
It compares how much overlap there is between your model’s generated summary and the reference (ground truth) summary.

Here’s what each score means:

ROUGE-1: Measures overlap of unigrams (individual words) between the generated and reference summaries.
→ Reflects how many key words your model captured.
ROUGE-2: Measures overlap of bigrams (two-word sequences).
→ Captures short phrase-level coherence and fluency.
ROUGE-L: Based on the Longest Common Subsequence (LCS) between generated and reference summaries.
→ Reflects how well the model preserves sentence structure and ordering.
In short:

Higher ROUGE-1/2 → Better lexical overlap (word and phrase choice).
Higher ROUGE-L → Better structural and semantic alignment.
Putting It All Together
Finally, the main evaluation orchestrates everything:

def evaluate_baseline():
    _, val_data, _ = load_and_prepare_dataset(cfg)
    model, tokenizer = setup_model_and_tokenizer(cfg, use_4bit=False, use_lora=False)

    preds = generate_predictions(
        model=model,
        tokenizer=tokenizer,
        dataset=val_data,
        task_instruction=cfg["task_instruction"],
        batch_size=4,
    )

    scores = compute_rouge(preds, val_data)

    print("\n💾 Results:")
    print(json.dumps(scores, indent=2))
    return scores
