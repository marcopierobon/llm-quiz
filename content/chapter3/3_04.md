You've already built your baseline and evaluated GPT-4o-mini through managed fine-tuning.
Now it's time to take control — to fine-tune an open-weight model yourself.


In this lesson, you'll train Llama 3.2 1B-Instruct on the SAMSum dialogue summarization dataset using QLoRA — a combination of 4-bit quantization and LoRA adapters.


We'll reuse everything you built in the earlier lessons: dataset loading, configuration management, and evaluation functions.
Then we'll add the training pipeline — with masking logic, Trainer setup, and model checkpointing.


By the end, you'll have a reproducible training script, a set of LoRA adapters, and measurable performance gains over your baseline.


A note on goals: This lesson focuses on getting the mechanics right — building a working pipeline that runs end-to-end. We're not optimizing for peak performance yet. That comes next, when we add experiment tracking and hyperparameter search.

Configuration: Your Training Blueprint
Before we dive into the training code, let's look at what's driving it — the config.yaml file.

You've already seen the dataset section in earlier lessons. Now we'll use the rest of the configuration: quantization settings, LoRA parameters, and training hyperparameters.

Let's break down the new sections you're using in this lesson.

Quantization Settings
These control how the model loads in memory:

load_in_4bit: true
bnb_4bit_quant_type: nf4
bnb_4bit_use_double_quant: true
bnb_4bit_compute_dtype: bfloat16
What this does: Loads the model in 4-bit precision using NormalFloat4 (nf4) quantization. The model uses 4-bit weights during training but computes in bfloat16 for stability. Double quantization compresses the quantization constants themselves for extra memory savings.

Why it matters: A full-precision 1B model needs ~4GB of memory. With 4-bit quantization, you fit it in ~1GB — making fine-tuning possible on consumer GPUs.

LoRA Configuration
These define your adapter parameters:

lora_r: 8
lora_alpha: 16
lora_dropout: 0.1
target_modules: ["q_proj", "v_proj", "k_proj", "o_proj"]
What this does: Injects low-rank adapters with rank 8 into the query, key, value and output projection layers (i.e. all attention blocks in the LLaMA model). The alpha/r ratio (16/8= 2) controls the scaling of LoRA updates. Dropout helps prevent overfitting.

Why it matters: Instead of updating all 1 billion parameters, you're training only ~1-2 million parameters in the adapters. This is what makes fine-tuning fast and memory-efficient.

Training Hyperparameters
These control the optimization process:

num_epochs: 1
learning_rate: 2e-4
batch_size: 4
gradient_accumulation_steps: 4
sequence_len: 512
lr_scheduler: cosine
warmup_steps: 50
What this does: Trains for 1 epoch with a learning rate of 2e-4, using an effective batch size of 16 (4 × 4 gradient accumulation). Sequences are capped at 512 tokens. The cosine scheduler gradually reduces the learning rate after 50 warmup steps.

Why it matters: These are conservative settings that work well for most instruction fine-tuning tasks. You can experiment with these later, but they're a solid starting point.

Memory Optimization
Two settings keep memory usage low:

bf16: true
optim: paged_adamw_8bit
BFloat16 training uses 16-bit precision for activations and gradients — much faster than 32-bit with minimal quality loss.

Paged AdamW 8-bit reduces optimizer state memory by using 8-bit precision and paging optimizer states to CPU when needed.

Together, these let you train on GPUs with 16GB or less.

The big picture: This configuration gives you a production-ready training setup with minimal memory requirements and solid baseline hyperparameters. Everything is exposed in YAML so you can experiment later without touching code.

Now let's see how the training script uses these settings.

Fine-Tuning Implementation - Key Components
Your main fine-tuning logic now lives in the training script. Here's what's new:

preprocess_samples() — converts dialogue/summary pairs into tokenized chat sequences and applies assistant-only masking
train_model() — configures the Trainer API, runs training, and saves LoRA adapters
main() — loads config, initializes model, dataset, and starts training
Let's walk through each one.

1️⃣ Assistant-Only Masking
To train effectively on instruction data, you need to compute loss only on the assistant's tokens — the summary in this case.

Why? Because you don't want the model wasting capacity learning to predict the user's dialogue. You want it focused on generating high-quality summaries.

Here's how that works in code:

def preprocess_samples(examples, tokenizer, task_instruction, max_length):
    """Tokenize dialogues and apply assistant-only masking for causal LM."""
    input_ids_list, labels_list, attn_masks = [], [], []

    for d, s in zip(examples["dialogue"], examples["summary"]):
        # Build message sequences
        msgs_full = build_messages_for_sample({"dialogue": d, "summary": s}, task_instruction, include_assistant=True)
        msgs_prompt = build_messages_for_sample({"dialogue": d, "summary": s}, task_instruction, include_assistant=False)

        text_full = tokenizer.apply_chat_template(msgs_full, tokenize=False, add_generation_prompt=False)
        text_prompt = tokenizer.apply_chat_template(msgs_prompt, tokenize=False, add_generation_prompt=True)
        prompt_len = len(text_prompt)

        tokens = tokenizer(
            text_full,
            max_length=max_length,
            truncation=True,
            padding="max_length",
            add_special_tokens=False,
            return_offsets_mapping=True,
        )

        # Find where the assistant response starts by matching character positions
        # offset_mapping tells us which character positions each token came from
        # We find the first token that starts at or after prompt_len
        start_idx = next((i for i, (start, _) in enumerate(tokens["offset_mapping"]) if start >= prompt_len), len(tokens["input_ids"]))

        # Mask everything before the assistant response with -100 (ignored in loss)
        labels = [-100] * start_idx + tokens["input_ids"][start_idx:]

        input_ids_list.append(tokens["input_ids"])
        labels_list.append(labels)
        attn_masks.append(tokens["attention_mask"])

    return {"input_ids": input_ids_list, "labels": labels_list, "attention_mask": attn_masks}
The key insight: offset_mapping tells you which character positions each token came from. We tokenize the full conversation, find where the assistant's response starts in the character string, then mask everything before that point with -100 (which the loss function ignores).

This ensures the model learns to generate summaries — not waste capacity predicting the user's dialogue.

Without proper masking, your model will try to learn patterns from both the user input and the assistant response. That sounds reasonable until you realize:

The model will never generate user inputs in production. It only generates assistant responses.

Training it to predict user text wastes capacity and can actually hurt performance — the model learns patterns it'll never use and might start generating summaries that mimic dialogue structure instead of concise summaries.

Masking fixes this by focusing all learning on what the model will actually do: generate summaries.

2️⃣ Training Configuration
Now we configure the Trainer. This is where you specify batch sizes, learning rates, optimizer settings, and logging.

def train_model(cfg, model, tokenizer, train_data, val_data):
    """Tokenize datasets, configure Trainer, and run LoRA fine-tuning."""
    task_instruction = cfg["task_instruction"]

    print("\n📚 Tokenizing datasets...")
    tokenized_train = train_data.map(
        lambda e: preprocess_samples(e, tokenizer, task_instruction, cfg["sequence_len"]),
        batched=True,
        remove_columns=train_data.column_names,
    )
    tokenized_val = val_data.map(
        lambda e: preprocess_samples(e, tokenizer, task_instruction, cfg["sequence_len"]),
        batched=True,
        remove_columns=val_data.column_names,
    )

    collator = DataCollatorForLanguageModeling(tokenizer=tokenizer, mlm=False)

    args = TrainingArguments(
        output_dir="./outputs/lora_samsum",
        num_train_epochs=cfg["num_epochs"],
        per_device_train_batch_size=cfg["batch_size"],
        gradient_accumulation_steps=cfg["gradient_accumulation_steps"],
        learning_rate=cfg["learning_rate"],
        lr_scheduler_type=cfg.get("lr_scheduler", "cosine"),
        warmup_steps=cfg.get("warmup_steps", 100),
        bf16=cfg.get("bf16", True),
        optim=cfg.get("optim", "paged_adamw_8bit"),
        logging_steps=cfg.get("logging_steps", 25),
        save_total_limit=cfg.get("save_total_limit", 2),
        report_to="wandb",
    )

    trainer = Trainer(
        model=model,
        args=args,
        train_dataset=tokenized_train,
        eval_dataset=tokenized_val,
        data_collator=collator,
    )

    print("\n🎯 Starting LoRA fine-tuning...")
    trainer.train()
    print("\n✅ Training complete!")

    save_dir = "./outputs/lora_samsum/lora_adapters"
    model.save_pretrained(save_dir)
    tokenizer.save_pretrained(save_dir)
    print(f"💾 Saved LoRA adapters to {save_dir}")
A few settings worth noting:

bf16=True enables BFloat16 training for memory efficiency without losing much precision
paged_adamw_8bit reduces optimizer memory footprint — critical when working with limited VRAM
gradient_accumulation_steps simulates larger batch sizes by accumulating gradients across multiple forward passes before updating weights
The rest is standard Hugging Face Trainer configuration. It handles the training loop, logging, and checkpointing automatically.

3️⃣ Running the Training
Everything ties together in the main() entry point:

def main():
    cfg = load_config()

    # Load dataset
    train_data, val_data, _ = load_and_prepare_dataset(cfg)

    # Initialize quantized model with LoRA adapters
    model, tokenizer = setup_model_and_tokenizer(cfg, use_4bit=True, use_lora=True)

    # Train
    train_model(cfg, model, tokenizer, train_data, val_data)
You load config, prepare data, initialize the model with 4-bit quantization and LoRA, and kick off training.

Evaluating Your Fine-Tuned Model
Once training finishes, you'll want to evaluate your LoRA adapters to measure improvement.

The evaluation process looks almost identical to what you did with the base model — except now you're loading the fine-tuned adapters on top of the base model.

qlora-evaluation-workflow.jpeg

Here's how that works:

def evaluate_lora_model(cfg):
    """Load base model, attach LoRA adapters, and evaluate."""

    # Load base model (same as baseline evaluation)
    model, tokenizer = setup_model_and_tokenizer(cfg, use_4bit=True, use_lora=False)

    # Attach your fine-tuned LoRA adapters
    adapter_dir = "./outputs/lora_samsum/lora_adapters"
    model = PeftModel.from_pretrained(model, adapter_dir)
    model.eval()

    # Load validation data
    _, val_data, _ = load_and_prepare_dataset(cfg)

    # Generate predictions (same inference function as before)
    preds = generate_predictions(
        model=model,
        tokenizer=tokenizer,
        dataset=val_data,
        task_instruction=cfg["task_instruction"],
        batch_size=cfg.get("eval_batch_size", 4),
    )

    # Compute ROUGE scores (same as before)
    scores = compute_rouge(preds, val_data)

    return scores, preds
The only difference from baseline evaluation is this line:

model = PeftModel.from_pretrained(model, adapter_dir)
You load the base model in 4-bit precision, then use PeftModel.from_pretrained() to attach your trained adapters. Everything else — dataset loading, inference, ROUGE computation — reuses the exact same utilities you built earlier.
