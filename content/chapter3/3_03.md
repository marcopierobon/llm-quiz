You’ve already built a baseline with Llama 3.2 1B.
In the next lesson, you’ll fine-tune it yourself using LoRA in Colab.


Before that, let’s see what happens when we use a frontier model like GPT-4o-mini for the same SAMSum summarization task — first as-is, then after fine-tuning.


This gives us a clear comparison between open-weight and frontier models on the same dataset and shows how much managed fine-tuning can still improve a strong base model.
Along the way, you’ll also learn how to fine-tune frontier models if you ever need to.


Think of this as a curiosity-driven benchmark before we return to hands-on fine-tuning with open-weight models.

Why This Quick Detour?
When people think of fine-tuning, they picture notebooks, GPUs, and custom training loops.
That’s what we’ll do next — but it helps to see what happens when all that work is handled for you.

Companies like OpenAI and Google offer a managed fine-tuning service: you upload your data, specify the base model, and they handle the rest — infrastructure, optimization, scaling, and checkpointing.

It’s quick and reliable, though it comes with trade-offs:

Less control or visibility into model internals
Higher per-token costs
No access to intermediate checkpoints
This lesson shows you the workflow once — to understand the trade-offs before returning to self-hosted fine-tuning in Colab.

From Here On: Two Workflows to Compare
Now that you understand how a single OpenAI API call works, let’s scale things up — and divide our work into two clear workflows:

Workflow 1 — Baseline Evaluation:
Run inference on the base GPT-4o-mini model using the same SAMSum dataset and evaluation functions from the previous lesson.
Workflow 2 — Managed Fine-Tuning:
Fine-tune GPT-4o-mini using OpenAI’s hosted API and then re-evaluate it using the exact same pipeline.
This separation keeps things consistent: we’ll reuse the same dataset, prompt, and ROUGE calculation across both workflows — changing only the model and its training source.

Workflow 1: Evaluating the Frontier Model (GPT-4o-mini Baseline)
Let's start with the baseline evaluation workflow.

Understanding the Mechanics: One Example at a Time
When using OpenAI’s API, we send a list of chat messages that describe the task. Each SAMSum dialogue becomes a “conversation” for the model, which replies with its summary.

from datasets import load_dataset
from openai import OpenAI

client = OpenAI()
dataset = load_dataset("knkarthick/samsum")
val_data = dataset["validation"].shuffle(seed=42).select(range(200))

def build_user_prompt(dialog: str):
    prompt = (
        "You are a helpful assistant who writes concise, factual summaries of conversations. "
        "Summarize the following conversation in one clear sentence. "
        "If it can be shorter, make it as brief as possible.\n"
        f"## Dialog: {dialog}\n"
        "## Summary:\n"
    )
    return prompt

sample = {
    "dialogue": (
        "Jane: Hey, are we still on for lunch?\n"
        "John: Yes, but let's move it to 1 PM instead of noon.\n"
        "Jane: Works for me."
    )
}

messages = [{"role": "user", "content": build_user_prompt(sample["dialogue"])}]

response = client.chat.completions.create(model="gpt-4o-mini", messages=messages, temperature=0)
print(response.choices[0].message.content)
Output:

Jane and John agreed to reschedule lunch from noon to 1 PM.
This is the same mechanism we’ll now scale to hundreds of samples — generate predictions for all, then compute ROUGE scores against reference summaries.

Implementing the Baseline Evaluation
We can reuse the same helper functions from the previous lesson to load the dataset, compute ROUGE, and structure the prompts. These are:

load_and_prepare_dataset(cfg) — to load and sample SAMSum data
compute_rouge(predictions, samples) — to evaluate summary quality
build_messages_for_sample(sample, task_instruction) — to structure OpenAI prompts
We’ll add just two functions:

generate_openai_predictions() — runs threaded API inference across all samples
evaluate_openai_model() — wraps the evaluation workflow (generation + scoring + saving)
Now let’s bring in our two helper functions to automate this process.

Generating Predictions
def generate_openai_predictions(model_name, dataset, task_instruction, num_samples=None, max_workers=10, sleep_time=0.25):
    """Generate predictions from an OpenAI model using concurrent threads."""
    if num_samples is not None and num_samples < len(dataset):
        dataset = dataset.select(range(num_samples))
    total = len(dataset)
    preds = [None] * total

    def process_sample(idx, sample):
        messages = build_messages_for_sample(sample, task_instruction, include_assistant=False)
        try:
            response = client.chat.completions.create(model=model_name, messages=messages, temperature=0.2, max_tokens=128)
            return idx, response.choices[0].message.content.strip()
        except Exception as e:
            print(f"⚠️ Error on sample {idx}: {e}")
            return idx, ""

    with ThreadPoolExecutor(max_workers=max_workers) as executor:
        futures = [executor.submit(process_sample, i, dataset[i]) for i in range(total)]
        for i, future in enumerate(as_completed(futures), start=1):
            idx, output_text = future.result()
            preds[idx] = output_text
            if i % 10 == 0 or i == total:
                print(f"✅ Completed {i}/{total}")
            time.sleep(sleep_time)
    return preds
Running the Evaluation
def evaluate_openai_model(model_name: str, cfg: dict, limit: int = None, max_workers: int = 5):
    """Run evaluation on an OpenAI model (base or fine-tuned)."""
    print(f"\n🚀 Evaluating OpenAI model: {model_name}")
    _, val_data, _ = load_and_prepare_dataset(cfg)
    if limit:
        val_data = val_data.select(range(limit))
    preds = generate_openai_predictions(model_name, val_data, cfg["task_instruction"], len(val_data), max_workers, 0.1)
    scores = compute_rouge(preds, val_data)
    print("\nROUGE Summary:")
    print(f"ROUGE-1: {scores['rouge1']:.2%}, ROUGE-2: {scores['rouge2']:.2%}, ROUGE-L: {scores['rougeL']:.2%}")
    return scores, preds

GPT-4o-mini (Base)	39.97%	15.50%	31.28%

Workflow 2: Fine-Tuning GPT Models with OpenAI’s Managed API
Now that we’ve seen how to evaluate the base model, let’s move on to the second workflow — fine-tuning the same model using OpenAI’s hosted infrastructure.

This workflow mirrors what you’ve already done locally with Llama 3.2, except that OpenAI handles the GPU resources, training loop, and checkpoints for you.

What We’ll Reuse
We’ll reuse:

The same evaluation function (evaluate_openai_model)
The same instruction prompt and ROUGE metrics
The new logic we need here is to:

Convert our dataset to the OpenAI JSONL format
Upload it to the API
Launch and monitor a fine-tuning job
Evaluate the resulting fine-tuned model
Step 1 — Preparing and Uploading Data
OpenAI’s fine-tuning API expects data in JSONL format, where each record is a chat example with user and assistant messages.

from openai import OpenAI
import json

client = OpenAI()

def build_user_prompt(dialogue):
    return (
        "You are a helpful assistant who writes concise, factual summaries of conversations. "
        "Summarize the following conversation into a single sentence.\n"
        f"## Dialog:\n{dialogue}\n"
        "## Summary:\n"
    )

# Convert dataset to JSONL format
def convert_to_jsonl(dataset, out_path):
    with open(out_path, "w") as f:
        for sample in dataset:
            record = {
                "messages": [
                    {"role": "user", "content": build_user_prompt(sample["dialogue"])},
                    {"role": "assistant", "content": sample["summary"]}
                ]
            }
            f.write(json.dumps(record) + "\n")
    print(f"✅ Saved: {out_path}")

convert_to_jsonl(train_data, "samsum_train.jsonl")
convert_to_jsonl(val_data, "samsum_val.jsonl")
Now upload both files to OpenAI:

train_file = client.files.create(file=open("samsum_train.jsonl", "rb"), purpose="fine-tune")
val_file = client.files.create(file=open("samsum_val.jsonl", "rb"), purpose="fine-tune")

print("Training file ID:", train_file.id)
print("Validation file ID:", val_file.id)
Step 2 — Creating the Fine-Tuning Job
Next, create a fine-tuning job using gpt-4o-mini as the base model:

job = client.fine_tuning.jobs.create(
    model="gpt-4o-mini",
    training_file=train_file.id,
    validation_file=val_file.id,
    suffix="samsum-ft",
    hyperparameters={"n_epochs": 1, "learning_rate_multiplier": 1, "batch_size": 8}
)

print("✅ Fine-tuning job created:", job.id)
The job runs entirely on OpenAI’s infrastructure — no GPUs or Colab setup needed.
You can monitor progress directly from the OpenAI dashboard, where loss and accuracy curves update in real time.

Step 3 — Monitoring Training Progress
Monitoring Progress

You can check progress from your notebook or terminal using:

import time

while True:
    status = client.fine_tuning.jobs.retrieve(job.id)
    print(f"🕒 {status.status} | Trained tokens: {status.trained_tokens or 0}")
    if status.status in ("succeeded", "failed", "cancelled"):
        break
    time.sleep(30)
Once complete, the response will include the fine-tuned model ID, for example:

ft:gpt-4o-mini-2024-11-07:ready-tensor-inc:samsum-ft
Step 4 — Evaluating the Fine-Tuned Model
Finally, evaluate your fine-tuned model using the same helper functions from Workflow 1:

tuned_model_id = "ft:gpt-4o-mini-2024-11-07:ready-tensor-inc:samsum-ft"
scores, preds = evaluate_openai_model(tuned_model_id, cfg, limit=200)
We get the following results:

ROUGE Summary:
ROUGE-1: 55.72%, ROUGE-2: 32.80%, ROUGE-L: 47.77%
Results So Far
Here are our results on the SAMSum summarization benchmark (200-sample validation set):

Model	Type	ROUGE-1	ROUGE-2	ROUGE-L
Llama 3.2 1B	Open-weight base	35.12%	13.04%	27.26%
GPT-4o-mini	Frontier base	39.97%	15.50%	31.28%
GPT-4o-mini (fine-tuned)	Managed fine-tuned	55.72%	32.80%	47.77%
The comparison shows a clear progression:

The Llama 3.2 1B model performs reasonably well but remains limited by its smaller size.
The base GPT-4o-mini already demonstrates strong summarization quality without fine-tuning.
The fine-tuned GPT-4o-mini significantly improves on all ROUGE metrics, achieving nearly 47.77% ROUGE-L, showing the benefit of even light managed fine-tuning.
These results illustrate how model scale and fine-tuning approach affect summarization performance — giving you a clear benchmark before moving to self-hosted fine-tuning in the next lesson.

What's Next

It’s an efficient way to get quick improvements, but it comes with trade-offs: limited transparency, higher costs, and less control over what happens behind the scenes.
